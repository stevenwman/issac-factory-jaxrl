## Task 1.1 — env_step protocol research findings (M5a)

### `EnvBundle.env_step` signature

- **Args:** `(state, action)` — one positional env-state struct, one action array.
  - `state` is a struct with fields `.obs`, `.reward`, `.done`, `.info`. It is NOT bare obs. The test stub confirms: `StubEnvState(obs={"state": ..., "privileged_state": ...}, reward=..., done=..., info={"truncation": ...})`.
  - `action` is a `(num_envs, action_dim)` array, already clipped to `[-1, 1]` by `onpolicy_collect` before the call (`clipped_action = jnp.clip(action, -1.0, 1.0)` at line 104).
- **Returns:** same struct type as input — a new state with updated `.obs`, `.reward`, `.done`, `.info`.
  - Concrete shape: `state.obs` → `(num_envs, obs_dim)` array or a dict of arrays; `state.reward` → `(num_envs,)`; `state.done` → `(num_envs,)` float (terminated | truncated, Brax convention); `state.info["truncation"]` → `(num_envs,)` float.
- **Called:** once per timestep inside `lax.scan`, not batched across steps. The scan loops `num_steps` times, each iteration calls `env_step(env_state, clipped_action)` exactly once.
- **File:line evidence:** `onpolicy_collect.py:106` — `env_state = env_step(env_state, clipped_action)`

### Gym backend reference pattern

- **Closure structure:** `env_step` is a plain Python closure defined inside `make_gym_env_bundle`. It captures `vec_env` (gymnasium `VectorEnv`) by reference. Signature: `(state: GymState, action) -> GymState`. The closure converts JAX/numpy action via `np.asarray`, calls `vec_env.step(a_np)`, and wraps results back into a `GymState` dataclass.
- **Dict-obs handling:** When the underlying env returns dict obs (e.g. `{"state": ..., "privileged_state": ...}`), `_to_numpy` coerces each value to numpy. The `GymState.obs` field becomes a dict of numpy arrays. The `has_privileged` and `dict_obs` flags on `EnvBundle` are detected from this initial obs shape. The `env_step` closure itself does no key remapping — it returns whatever the gym env emits.
- **Key metadata on EnvBundle:** `has_privileged = dict_obs and "privileged_state" in obs`; `critic_obs_dim = obs["privileged_state"].shape[-1]` if privileged, else `None`. These drive the `get_policy_obs` / `get_critic_obs` lambdas in the training script.
- **File:line evidence:** `gym_backend.py:241-258` (closure), `gym_backend.py:264-276` (dict-obs detection), `gym_backend.py:285-295` (EnvBundle construction)

### RolloutBatch / RolloutBuffer

- **Asymmetric critic_obs:** `RolloutBatch` does NOT have a `critic_obs` field. The `StepData` NamedTuple (inside `onpolicy_collect`) carries both `obs` (normalized policy obs) and `critic_obs` (normalized privileged obs) per step. The training script passes `rollout.critic_obs` directly as the `critic_obs` kwarg to `ppo.update()`, bypassing `RolloutBatch` entirely for the critic dimension. `RolloutBatch` only holds actor-side `obs`.
- **GAE:** Computed INSIDE `PPO.update` (specifically inside `_update`'s `epoch_step`), NOT by the caller. The batch passed to `ppo.update()` does NOT need `advantages` or `returns` pre-computed. GAE is re-run per epoch from fresh critic values — matches Brax's per-epoch recomputation. The `RolloutBatch.advantages` / `RolloutBatch.returns` fields exist on the dataclass but are never populated by the `onpolicy_collect` flow; they are `None` when passed to `update()`.
- **Bootstrap value at next_obs:** `onpolicy_collect` computes `next_value` (the bootstrap value after the last step) using `select_deterministic` on `normed_next` / `normed_next_critic` and returns it. The training script passes this as `next_obs=normed_next` to `ppo.update()`. Inside `_update`, `critic.apply(state.critic_params, critic_next_obs)` re-derives the bootstrap value from the normalized next critic obs each epoch.
- **File:line evidence:** `buffers/rollout.py:8-21` (RolloutBatch fields), `onpolicy_collect.py:40-48` (StepData with critic_obs), `algos/ppo.py:138-171` (_update with per-epoch GAE), `train_ppo_contraction.py:277-295` (batch construction + ppo.update call pattern)

### Implications for our backend.py

- **env_step closure must return:** a struct (NamedTuple or dataclass) with `.obs`, `.reward`, `.done`, `.info` where `.info["truncation"]` is always populated. The IsaacLab backend should define an `IsaacLabState` mirroring `GymState`.
- **env_state in EnvBundle:** set to the initial `IsaacLabState` after env reset. The `.obs` field should be a dict `{"state": <actor_obs>, "privileged_state": <privileged_obs>}` when asymmetric critic is in use (spec D8 — remapping from Isaac's `policy` and `critic` keys happens inside `env_step` or in the initial state construction, NOT in `onpolicy_collect`).
- **Pre-compute next_obs / next_critic_obs:** No. `onpolicy_collect` reads `env_state.obs` after the last scan step and calls `get_policy_obs` / `get_critic_obs` on it directly. The backend only needs to return correct `.obs` in every `env_step` return; bootstrap obs extraction is handled by the collect loop.
- **Obs key remapping (policy->state, critic->privileged_state per spec D8):** Goes INSIDE `env_step`. Isaac returns `obs_dict["policy"]` and `obs_dict["critic"]`; `env_step` should remap these to `{"state": ..., "privileged_state": ...}` in the returned `IsaacLabState.obs`. `onpolicy_collect` and `obs_pipeline` only ever read `env_state.obs["state"]` and `env_state.obs["privileged_state"]`.

### Open follow-ups (if any)

- `RolloutBatch` has no `critic_obs` field, but `ppo.update()` accepts `critic_obs` as a separate kwarg. The training script passes `rollout.critic_obs` (from `StepData`) directly. Our backend.py / training script must follow the same pattern — do NOT try to pack `critic_obs` into `RolloutBatch`.
- The `done` flag semantics: `GymState.done = terminated | truncated` (float). Isaac's `ManagerBasedRLEnv.step()` returns `terminated` and `truncated` as separate tensors — we must OR them into `done` and store `truncated` separately under `info["truncation"]`. Confirm this at M5a implementation time.
- Whether IsaacLab's dlpack bridge introduces per-step latency that breaks the `lax.scan` assumption of pure JAX operations: the gym backend is pure Python (scan carries numpy arrays, not JAX). The isaaclab backend will be the same pattern — `env_step` is a Python closure, env state is numpy/dlpack-converted, scan operates on JAX-compatible arrays. No blocker here, but verify dlpack zero-copy on first smoke test.

## Task 1.3 — Isaac DirectRLEnv reset/step semantics findings (M5a)

### DirectRLEnv.reset

- **Signature:** `def reset(self, seed: int | None = None, options: dict[str, Any] | None = None) -> tuple[VecEnvObs, dict]`
- **Seed kwarg supported:** yes. When `seed is not None`, calls `self.seed(seed)` before resetting all envs.
- **Options kwarg used:** accepted for Gymnasium compatibility but NOT used internally — no conditional logic branches on `options`. Effectively a no-op passthrough.
- **Reset scope:** resets ALL environments unconditionally (calls `_reset_idx(torch.arange(self.num_envs))`). This is a full reset, not selective.
- **Returns:** `(obs_dict, extras_dict)` — 2-tuple, NOT the 5-tuple from step.
- **File:line evidence:** `direct_rl_env.py:292-330`

### DirectRLEnv.step

- **Signature:** `def step(self, action: torch.Tensor) -> VecEnvStepReturn`
- **Return tuple shape:** 5-tuple: `(obs, reward, terminated, truncated, info)`
  - Type alias: `VecEnvStepReturn = tuple[VecEnvObs, torch.Tensor, torch.Tensor, torch.Tensor, dict]`
  - `obs`: `dict[str, Tensor]` — after auto-reset (see below)
  - `reward`: `(num_envs,)` tensor — the reward earned at the CURRENT step (before reset)
  - `terminated`: `(num_envs,)` bool tensor — `reset_terminated`
  - `truncated`: `(num_envs,)` bool tensor — `reset_time_outs`
  - `info`: `extras` dict
- **File:line evidence:** `direct_rl_env.py:333` (signature), `envs/common.py:101` (type alias), `direct_rl_env.py:430` (return statement)

### Auto-reset behavior

- **At a done-step, returned obs is: POST-RESET INITIAL STATE** (not terminal state).
- The step sequence inside `step()` is:
  1. Physics forward pass
  2. `_get_dones()` → populate `reset_terminated` and `reset_time_outs`
  3. `_get_rewards()` → populate `reward_buf` (reward is computed BEFORE reset)
  4. `_reset_idx(reset_env_ids)` — resets all done envs IN-PLACE
  5. `_get_observations()` → obs computed AFTER reset
  6. Return `(obs_buf, reward_buf, reset_terminated, reset_time_outs, extras)`
- Consequence: the `obs` field in the returned tuple for a done env is the first obs of the NEW episode, not the final terminal obs. The terminal observation is never directly returned. Reward IS the terminal reward (from step 3, before reset).
- This matches the standard gymnasium auto-reset contract where `obs` at done is `reset_obs`. The true terminal obs is NOT available in any return field — it's discarded.
- **File:line evidence:** `direct_rl_env.py:395-430` (step body order of operations, especially `_reset_idx` before `_get_observations`)

### Factory NutThread specifics

- **Episode termination signal:** There is NO task-success-based early termination in Factory. `_get_dones()` returns `(time_out, time_out)` — both `terminated` and `truncated` are the SAME tensor. Done iff `episode_length_buf >= max_episode_length - 1`. There is no "task succeeded → terminate" logic; all episodes run to the time limit.
  - Concretely: `reset_terminated = time_out`, `reset_time_outs = time_out`. At the step() return, `terminated == truncated` always.
- **Truncation (timeout) handling:** NOT separately signaled from termination — both fields are identical. Factory's `_get_dones` explicitly returns `(time_out, time_out)`. From the jax-learning perspective, `info["truncation"]` should be set from `truncated` (which equals `terminated`), but since they're the same, the practical effect is that every episode end is a timeout/truncation, never a genuine terminal transition. Bootstrapping in GAE should use `truncated` mask properly (Task 1.1 confirmed `handle_truncation=True` in TrainConfig — this is correct and important here).
- **Episode length in cfg:**
  - Config field: `FactoryTaskNutThreadCfg.episode_length_s = 30.0` (seconds)
  - `decimation = 8`, `sim.dt = 1/120 s`
  - `max_episode_length = ceil(30.0 / (8 * (1/120))) = ceil(30.0 / 0.0667) = ceil(450.0) = **450 steps**`
  - This resolves the Task 1.2 unknown: `TrainConfig.episode_length` should be set to **450**.
  - File:line: `factory_env_cfg.py:206-209` (NutThread class), `factory_env_cfg.py:72,97` (decimation and sim.dt)
- **Reward at terminal step:** Reward is computed normally via `_get_rewards()` before the reset. No special terminal reward or reward shaping at episode end is visible in the base Factory env. Task-specific reward computation may add bonuses (e.g. success rewards) — not verified here at the task level, but the FRAMEWORK doesn't suppress/modify reward at done steps.
- **Success tracking:** Factory tracks success in `extras["successes"]` and `extras["success_times"]` via `_log_factory_metrics()`, but these are METRICS only, NOT termination conditions.

### Implications for our backend.py

- **`env_step` closure construction of `IsaacLabState.info["truncation"]`:** take directly from the `truncated` tensor returned by `env.step()` (4th element of the 5-tuple). For Factory specifically, `truncated == terminated` always, but our code should still read from `truncated` for correctness/generality.
- **`env_step` closure construction of `IsaacLabState.done`:** `done = terminated | truncated` (float cast). For Factory this collapses to just `terminated` (they're identical), but OR-ing is correct as a general pattern matching the Brax/GymState contract from Task 1.1.
- **Whether to call `env.reset()` ourselves between iterations:** NO. The env auto-resets done envs inside `step()` before returning observations. Calling `env.reset()` between training iterations would redundantly reset all envs including ones mid-episode. Only call `env.reset()` once at startup to get the initial `IsaacLabState`.
- **TrainConfig `episode_length` value for NutThread:** **450** (computed above).
- **Obs at done-step is post-reset:** This is the correct behavior for the GAE bootstrap — after the last scan step, `env_state.obs` is always a valid "next obs" (either continuing-episode obs or post-reset obs for done envs). `handle_truncation=True` in TrainConfig + `info["truncation"]` mask ensures GAE bootstraps correctly at truncated boundaries. No special handling needed in our backend for the terminal-obs problem.

### Open follow-ups

- The `terminated == truncated` equality in Factory `_get_dones` means we can NEVER distinguish genuine task failure from timeout in the GAE computation. This is an upstream design choice (Factory has no early-stop on failure). No action needed for our backend, but worth noting for reward-shaping discussions.
- Terminal obs is discarded by auto-reset. If we ever need the true terminal obs (e.g. for a terminal value bootstrap in a non-truncation-masking PPO variant), we would need to cache it before `_reset_idx` runs — not feasible without modifying DirectRLEnv. With `handle_truncation=True`, the current setup is correct without needing terminal obs.
- Task-level reward shaping (NutThread-specific bonus at success) not examined here. If `_get_rewards` in `factory_nut_thread.py` includes a success bonus, it fires at terminal step normally (reward is computed before reset). No concern for correctness, but may want to verify reward magnitude scale at M5a implementation.
